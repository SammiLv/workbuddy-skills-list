# 钉钉个人一周工作总结

## 适用场景

当用户说“帮我总结钉钉一周的工作”或表达类似需求时，使用本 skill。目标是围绕当前用户一周内真实发生的工作证据，提炼“推动了什么、解决了什么、形成了什么结果、下一步还要跟进什么”，生成一份可直接用于向管理层汇报的个人工作总结。

本 skill 是通用型个人周报流程，不固定为某个姓名。每次运行都必须先通过 dws 查询当前用户的钉钉身份信息，确认总结对象后再采集证据。不要从群聊中频繁出现的姓名、会议参与人或文档作者中猜测总结对象。

## 触发与输出方式

- 用户输入“帮我总结钉钉一周的工作”，且提示词后没有其他参数时，在当前会话直接输出总结。
- 用户输入“帮我总结钉钉一周的工作 + 钉钉文档链接”时，将最终内容写入用户指定的钉钉文档或钉钉文件夹。
- 如果用户明确要求只输出台账、只写文档、补充来源限制、指定日期范围或指定某个人，则按用户要求执行，并在结果中写清楚实际范围和对象。

## 时间范围

默认时间范围为当前日期起向前回溯 7 天，使用 Asia/Shanghai 日期。输出中必须写明精确日期范围，例如“2026年5月20日-2026年5月26日”。

如果用户明确说“本周”“上周”或给出具体起止日期，优先使用用户指定的范围，并在最终结果中写清楚。

## 数据源范围

优先使用只读方式采集证据。除非用户明确要求写入钉钉文档，否则不要创建、修改、删除或发送任何钉钉数据。

全部钉钉数据采集都必须通过 dws 完成，包括联系人、日程、会议纪要、文档、表格、AI 表格、群聊、单聊、@我和我个人聊天窗口 信息。不要改用非 dws 方式采集数据。

不要采集、汇总或引用待办、OA 审批和钉钉日志。即使 dws 暴露相关能力，也不要把待办、OA 审批或日志作为本 skill 的证据来源；如果其他来源中偶然出现审批或待办字样，只能在与核心业务推进直接相关时作为聊天/文档上下文概括，不按审批或待办来源展开。

每次运行尽量覆盖以下来源：

1. 联系人：查询当前用户身份、姓名、userId，以及证据中涉及的关键联系人。
2. 日程与会议纪要：采集一周内的会议、评审、培训、项目讨论、跨部门对齐等事件；对能通过 dws 读取到的会议纪要、AI 听记、会议摘要、行动项和待跟进事项，必须进一步提炼关键决策、风险、责任分工、后续动作和需要当前用户推动的内容。不要把“参会”本身写成主要成果。
3. 文档、表格、AI 表格：围绕项目、会议纪要、聊天中出现的项目名、会议名、客户名、系统名、风险点和行动项，查找能解释工作内容、状态、结果、风险和决策的材料。
4. 群聊、单聊、@我和我个人聊天窗口：采集与当前用户相关的决策、任务、进度、问题、承诺和后续动作。

## 群聊和单聊采集规则

必须通过 dws 采集群聊、单聊、@我和我个人聊天窗口 证据。不要因为某类 dws 数据没有返回结果就静默跳过聊天来源；应继续使用 dws 中可用的群聊、个人聊天、机器人消息、DING 消息或相关检索能力交叉查找。

采集聊天证据时：

- 优先查看 @当前用户、最近单聊、近期工作群、项目群、会议相关群。
- 使用日程、会议纪要、文档、表格、AI 表格、群聊和单聊中发现的项目名、会议名、客户名、系统名、风险点和行动项作为检索关键词。
- 只纳入可归属到当前用户的工作证据，包括本人推动的事项、本人作出的决策或判断、本人明确要求他人推进的事项、本人被 @ 或被点名负责的事项、本人承诺跟进的事项。
- 私聊内容只做工作事实概括，不长篇引用原文，不输出无关隐私内容。
- 如果 dws 无法访问某类聊天数据、搜索结果受限或聊天记录不可读，必须在“未覆盖/受限来源”中说明。

## 证据判断

采集后先整理证据台账，再写总结。每条可用证据建议记录为：

`日期｜事项｜来源｜归属｜结果`

来源写法保持简洁，例如：

- `钉钉日程 / 会议标题`
- `会议纪要 / 会议标题`
- `文档 / 文档名`
- `表格 / 表格名`
- `AI表格 / 表名`
- `群聊 / 群名`
- `单聊 / 人名`

只保留能说明实际工作的证据。忽略纯通知、系统消息、签到、步数、普通提醒、无业务含义的抄送和无法归属到当前用户的泛泛讨论。

如果同一事项出现在多个来源中，合并为一条工作事项，并保留主要来源；如果来源之间存在冲突，优先级为：会议纪要/AI 听记、日程详情、文档、表格、AI 表格、聊天概括。无法确认的地方要写明不确定性。

## 分类规则

最终按五个固定板块输出，其中第 0 部分是概览，第 1-4 部分是工作分类。每条工作只能归入一个最合适的分类，不要跨板块重复。

0. 本周概览：用 3-5 句话概述本周管理重点、关键推进、问题处理、组织协同、阶段结果和下一步关注点。
1. 发现与解决问题：排查、风险识别、问题处理、流程优化、纠偏、故障或阻塞解决。
2. 业务与培训：需求推进、交付上线、客户支持、业务材料、内容建设、培训落地或能力建设。
3. 管理与协作：重点项目推进、跨部门对齐、任务闭环、资源协调、风险管控、项目节奏管理。
4. 学习与创新：工具探索、AI 应用、流程自动化、新方法沉淀、可复用模板或效率改进。

分类时以事项的主要工作性质为准。会议本身不是成果，不能因为日程中出现会议就写“参与会议”；要从会议纪要、后续聊天和相关材料中提炼会议推动的决策、问题、责任分工、行动项和业务结果。会议推动的需求交付可归入“业务与培训”；会议中发现并解决的风险可归入“发现与解决问题”；会议形成的管理机制、节奏控制或跨部门闭环可归入“管理与协作”。

## 输出格式

默认标题：

`钉钉一周工作总结`

正文结构、章节顺序、证据台账和“未覆盖/受限来源”的模板统一以 `references/template.md` 为准。`SKILL.md` 只维护采集范围、判断规则、写作口径和发布规则，不再内置另一套正文模板；如果两者冲突，以 `references/template.md` 的输出骨架为准，同时遵守本文件的数据源边界和质量要求。

## 写入钉钉文档

当用户附带钉钉文档链接或明确要求写入钉钉文档时：

1. 判断链接类型：如果是具体文档链接，优先将内容写入该文档；如果是文件夹或节点链接，在该位置创建总结文档。
2. 文档标题固定使用 `钉钉一周工作总结`，不随姓名或日期变化；只有用户明确指定标题时才按用户指定标题执行。
3. 如果是在文件夹中创建文档，先列出目标文件夹，查找完全同名文档；如存在同名文档，先删除同名文档，再创建新文档。
4. 如果当前会话没有删除文档工具，或删除失败，不要创建带后缀的重复文档；停止写入并向用户说明阻塞。
5. 创建或更新后读取回文档内容，确认标题、概览和四个板块存在，再向用户返回文档链接。

## 质量要求

- 全文使用中文，语气正式，可直接用于向管理层汇报。
- 输出不是流水账，不按日程逐条罗列；要从证据中提炼管理动作、业务影响、阶段结果、风险判断和后续抓手。
- 写具体工作，不写泛泛的“使用钉钉较多”“沟通频繁”等统计性描述。
- 尽量包含项目名、会议名、交付物、决策、风险、结果、后续状态和需要继续推动的事项。
- 每个要点先写管理动作和结果，再补充协作对象或来源背景。
- 少用“参与”“参会”“整理议题”“处理日常安排”等流水账表述；优先使用“推动、推进、明确、拉齐、统筹、协调、闭环、识别、解决、沉淀、建立、优化”等管理动词。
- 不把“参加会议”“主持会议”“整理议题”单独作为向上汇报重点，除非同时写清该会议沉淀的决策、行动项、风险处置或业务结果。
- 不长篇引用聊天原文，不输出与周报无关的隐私内容。
- 不把同一工作事项重复放入多个板块。
- 证据不足时不要编造结论，应写成“未发现明确结果”或放入受限来源说明。
- 读不到聊天历史时，必须说明 dws 的具体限制，不要改用非 dws 方式补采。

## 唯一输出模板

生成正文前必须读取并使用 `references/template.md`。不要在 `SKILL.md` 中复刻或临时改写另一套模板。
