# 钉钉组长例会议题

## 1. 目标

使用 `dws` skill 从本周周一到周五的钉钉群沟通和个人聊天记录中，整理需要在组长例会上沟通、确认、推进或升级的少量议题，并通过钉钉单聊发送给当前用户本人。

除非用户明确改口，不要创建钉钉文档、钉盘文档、普通周报文档，也不要写入日程会议纪要；输出目标必须是当前用户本人的钉钉单聊消息。

## 2. 默认范围

- 时区：`Asia/Shanghai`
- 沟通观察范围：本周周一 `00:00` 到本周周五 `23:59:59`
- 数据源范围：默认同时筛选工作群沟通和当前用户个人聊天记录；个人聊天只提炼与组长例会相关的议题，不写入私人链接、账号、密钥、无关闲聊或敏感原文
- 消息接收人：当前登录的钉钉用户本人，通过 `dws contact user get-self --format json` 获取，不要猜测 userId
- 消息标题：默认 `组长例会议题`
- 默认身份：当前用户本人；不要把群里高频出现的人名误当成当前用户

如果用户给了明确日期范围、指定群或指定消息标题，优先使用用户提供的信息。

## 3. 工具使用原则

必须先使用 `dws` skill 作为钉钉能力入口，不要绕过 `dws` 直接调用 HTTP API、浏览器或旧的单产品 MCP。使用前先读取 `/Users/sammilv/.codex/skills/dws/SKILL.md`，并遵守其中的 `--format json`、schema 优先、危险操作确认和错误处理规则。

本 skill 涉及的 `dws` 产品优先级：

- 群聊和个人聊天证据：`dws chat`
- 当前用户身份确认：`dws contact`
- 发送最终议题消息：优先 `dws chat message send-direct`，必要时才回退到 `dws chat message send`

### 3.1 群沟通和个人聊天证据

先按 `dws` skill 要求读取 `chat` 产品参考，并用 `dws schema` 或 `dws chat ... --help` 确认当前可用命令和参数，所有读取命令都加 `--format json`。

- 优先用 `dws chat message list-all --start ... --end ... --format json` 按本周时间范围拉取当前用户所有会话消息，再按会话类型、会话名、关键词、当前用户本人发言和用户被提及情况筛出工作群与个人聊天证据。
- 如果用户指定群名，先用 `dws chat search --query ... --format json` 找到 `openConversationId`，再用 `dws chat message list --group ... --time ... --format json` 读取该群消息。
- 如果需要补充用户被点名事项，可用 `dws chat message list-mentions --start ... --end ... --format json`。
- 如果需要按关键词补充检索，可用 `dws chat message search --keyword ... --start ... --end ... --format json`。
- 如果 `list-all` 或等价消息读取命令不可用，说明消息源受限，并询问用户是否提供群名、截图、导出的聊天记录，或允许使用桌面 UI。
- 如果需要使用 Computer Use 读取钉钉桌面聊天记录，先确认用户允许，因为这会读取实际聊天界面；用户已授权的情况下，可以同时查看群聊和个人聊天列表。
- 读取证据阶段不要发送任何消息、建群、改群名、改群成员或触发机器人通知。

### 3.2 发送给当前用户

使用 `dws contact` 获取当前用户身份，再把最终议题内容发给当前用户本人的钉钉单聊；所有命令都加 `--format json`，发送前用 `dws schema` 或 `--help` 确认字段。

1. 用 `dws contact user get-self --format json` 获取当前登录用户信息，提取 `userId`；如果返回 `openDingTalkId` 且当前 `message send` schema 更适合使用它，也可以使用 `--open-dingtalk-id`。
2. 优先确认 `dws chat message send-direct` 的实际参数，以 `--help` / `dws schema` 为准；该命令更接近普通单聊文本气泡的阅读宽度。
3. 优先使用 `dws chat message send-direct --user <userId> --title "组长例会议题" --text "<最终议题内容>" --format json` 发送给当前用户本人；如果当前运行时不支持 `send-direct`，再回退到 `dws chat message send --user <userId> --title "组长例会议题" --text "<最终议题内容>" --format json`。
4. 发送后检查返回结果是否成功；如果失败，按 `dws` skill 的错误处理规则重试或报告错误。
5. 不要把最终议题发到群聊、机器人、Webhook、DING 电话/短信或钉钉文档。

## 4. 议题筛选规则

只保留适合和组长沟通的事项，必须简单、去重、可讨论。优先级从高到低：

1. 需要组长拍板、协调资源、推动跨组协作的事项。
2. 群里出现的阻塞、风险、延期、口径不一致、依赖未闭环。
3. 当前用户本人发出的工作思考、方案判断、待确认事项、管理建议或对组长/团队协作有参考价值的记录。
4. 需要在例会上同步进展、结果、下周计划的重点项目。
5. 可以沉淀为流程规范、复盘改进、人员协作规则的问题。
6. 个人聊天中由用户自己记录、提醒或转发的本周工作议题，例如运营推广类需求策略、方案设计方法、工具使用成本对比、会议压缩、流程调整、跨组协调、待组长确认的个人想法；这类内容要转写成可讨论议题，不要把私人聊天原文照搬进消息。

剔除以下内容：

- 纯通知、闲聊、表情、打卡、系统消息。
- 当前用户只是被 @ 到、只需要知悉或参会，没有明确问题、决策、风险、待办或需协调动作的内容；例如普通会议通知、版本需求会通知、群公告。
- 与当前用户职责无关，且不需要组长介入的群信息。
- 已经明确闭环且不需要例会同步的小事项。
- 无法从证据中判断为工作事项的猜测。
- 个人聊天中的私人链接、账号、密钥、订阅地址、个人隐私和与工作无关内容。

去重要求：

- 同一事项在多个群、单聊或多条消息里重复出现时，只保留一条议题。
- 如果一个主题包含“方案、排期、负责人、风险”多条消息，合并为一个议题，不要拆成多个小节。
- 一般只输出 3-6 条最值得讨论的议题；除非用户明确要求完整清单，不要把所有工作动态都列出来。

## 5. 发送内容格式

发送给当前用户的钉钉消息优先使用接近普通文本的中文格式，简单明了，保留最少来源线索，不要大段引用聊天原文，不要按“待决策/风险/进展”额外分类。

为避免钉钉把消息渲染成窄版 Markdown 卡片，正文格式必须遵守：

- 标题使用普通文本 `本周需与组长沟通议题`，不要使用 `#` / `##` Markdown 标题。
- 列表使用普通项目符号 `•`，不要使用 Markdown 列表符 `-`。
- 不要使用多级缩进、引用块、表格、代码块或分割线。
- 每条议题控制为一段自然换行文本，尽量保持和手工发送的普通文本气泡一致。

推荐结构：

```text
本周需与组长沟通议题

• 运营推广类需求的方案设计策略需要统一：产品页、活动、广告、增值服务、C端产品等推广需求，是否先用 AI 产出思路和初步方案，再进入业务评审。来源：个人聊天 / 05-28
• ...
```

每条议题用一句话说清楚“要讨论什么 + 为什么需要讨论/确认”，可在句末追加 `来源：群名或个人聊天 / 日期`。不要写成多层嵌套，不要展开长背景。

## 6. 发送方式

发送前先整理完整消息正文：

- 如果没有筛出有效议题，也发送一条简短消息说明本周未发现需要组长例会沟通的明确议题，并注明覆盖的日期范围。
- 如果消息长度有限，保留高优先级议题，删减来源描述，不要删除议题本身。
- 发送前做一次人工复核：删除仅通知/仅知悉/仅被 @ 的内容，补充当前用户本人发出的可讨论工作想法，合并重复议题。
- 如果发送失败，不要改用群聊、机器人或钉钉文档作为替代；在当前会话中说明失败原因，并展示待发送内容。

发送后最终回复用户时说明：

- 覆盖的沟通日期范围
- 使用的 `dws` 数据源和发送方式
- 接收人是当前用户本人
- 发送议题数量
- 未覆盖的数据源或受限原因

## 7. 安全边界

- 默认只用 `dws` 读取群沟通和个人聊天，最后只向当前用户本人发送一条钉钉单聊消息。
- 不要创建钉钉文档作为替代输出。
- 不要向群聊发送任何内容，不要使用机器人或 Webhook 发送。
- 不要创建、删除、取消或修改日程。
- 不要擅自修改会议时间、参会人、会议室或循环规则。
- 如果发送给当前用户失败，向用户说明失败原因，并把待发送议题内容直接展示在会话中。
